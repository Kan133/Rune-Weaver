# QA

## Q1. Rune Weaver 这个想法可靠吗？

可靠，但前提是目标定义准确。

Rune Weaver 当前可行的方向不是“让小白一句话做任意复杂游戏”，而是：

> 在宿主 API 严格、玩法结构可复用、代码边界清晰的游戏垂类里，用 `schema + blueprint + pattern + adapter` 把中低复杂度内容稳定生成为宿主可消费代码。

## Q2. 依托 `pattern + LLM + blueprint`，真的能让小白上手吗？

可以，但条件是：

1. 宿主必须被强约束
2. 输出结果必须可感知
3. 生成边界必须受控

对当前 Dota2 路线来说，“小白上手”真正依赖的是：

- `init -> create -> run` 足够顺
- 结果能在游戏里直接看见
- 不要求用户理解宿主内部结构

## Q3. 能产出足够 robust 的代码吗？

能，但范围要限定。

### 可较稳定的部分

- 骨架级代码
- host bridge
- wiring
- TSX / LESS 模板
- 中低复杂度 mechanic logic

### 暂不应承诺的部分

- 复杂长尾逻辑
- 高度定制跨系统耦合
- 大量自由 UI 行为
- 任意宿主旧文件智能改写

结论：

Rune Weaver 可以做出“对一部分中低复杂度玩法足够 robust 的代码生成系统”，但短期不能承诺“通吃所有游戏逻辑”。

## Q4. 为什么 Dota2 / Roblox 这类宿主更适合？

因为这类宿主通常具有：

- API 严格
- 输出面有限
- 事件驱动明显
- UI 宿主清楚
- 玩法结构可重复

这正适合：

- pattern 化
- blueprint 化
- adapter 化
- validator 前置

## Q5. Rune Weaver 当前最现实的定位是什么？

当前最现实的定位是：

> 一个面向严格宿主、有限机制、短流程玩法的受控 `NL-to-Code` 系统。

它不是：

- 通用 AI 游戏开发神机
- 任意自然语言直接生成整游戏
- 任意宿主任意代码都能改的代理

## Q6. 先做 Dota2，后做 Roblox / Unity / UE 的路线是否成立？

成立，但节奏必须现实。

### Dota2

是合理的第一宿主。

### Roblox

是合理的第二宿主候选。

### Unity / UE

属于更远期方向，不应作为当前路线成立与否的判断标准。

## Q7. Rune Weaver 当前真正应该验证什么？

当前应该验证：

1. 自然语言能否稳定转成 `IntentSchema`
2. 不同题材需求能否复用相同 mechanic patterns
3. `Blueprint -> AssemblyPlan -> Host` 是否能形成受控代码落地链路
4. 普通用户能否完成一次最小闭环

## Q8. 当前不该承诺什么？

当前不该承诺：

- 任意复杂需求都能自动生成
- 任意 UI 个性化都能自由处理
- 任意宿主都能平滑支持
- 任意历史手写代码都能智能 merge

## Q9. 当前最危险的产品误区是什么？

1. 过早把自己想成通用引擎平台
2. 过早把 UI 个性化交给自由生成
3. 过早承诺“任意需求都能生成”

## Q10. 当前最合理的工程目标是什么？

当前最合理的工程目标是：

> 先证明 Rune Weaver 能在 Dota2 这种强宿主约束环境下，把一批中低复杂度、可复用骨架的玩法需求，稳定转成可运行、可感知、可验证的代码输出。

## Q11. 现实可做 / 暂不承诺 / 远期方向

### 现实可做

- Dota2 宿主下的中低复杂度 mechanic system
- 受控 server/shared/ui/bridge 代码生成
- pattern 驱动的组合式输出
- 对普通用户可感知的最小闭环

### 暂不承诺

- 任意复杂玩法系统
- 任意宿主旧文件智能改写
- 完整 UI 设计自动化
- code-level gap fill 作为主干能力

### 远期方向

- Roblox 宿主
- 更复杂的 UI Wizard
- 更广泛的 constrained gap fill
- Unity / UE 等复杂引擎
