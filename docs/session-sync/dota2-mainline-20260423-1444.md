# Current blockers / risks

- The former product/live blocker is closed: bridge artifact, connected-host status, workspace store, and feature review UI are read-model-first when `governanceReadModel` exists.
- The front-door create blocker is also closed: `equipment` no longer drifts to exploratory because of wizard/front-door leakage, and ambiguous weighted-card prompts no longer slip into write-path fallback.
- Compatibility-only fallback still exists for stale host-status, raw workspace, legacy bridge, and legacy workbench-result payloads. That fallback is deliberately display-only and must not grow back into product authority.
- `export-bridge` is the only refresh lane for those stale payloads. `doctor`, `validate`, and repair flows may inspect or prove honesty around the same truth, but they do not refresh legacy payloads.
- `repairability` is intentionally observational. Connected-host status and `workbench --inspect` build the governance read-model without live validate/repair observation, so `Repairability: not_checked` is expected there and should not be misread as missing canonical truth.
- The `grant_only_provider_export_seam` remains admitted as a Dota2 `seam`, but that admission does not change provider feature lifecycle, reveal-only lifecycle, or any runtime semantics.
- The remaining fresh risk exposed by proof is same-host trigger collision when stacking multiple `F4`-bound local selection features. That is an honest governance/integration issue after Blueprint/family closure, not a Stage 1 wizard problem.
- Dota2 owns the builder on purpose. Do not upscope this read-model into core until another host proves the same axes without needing Dota2-specific seam or doctor semantics.
- Step 7 closure is now policy-driven rather than “decide later”: fallback shrink, refresh cadence, and no-core genericization all have fixed guards and corresponding tests.
