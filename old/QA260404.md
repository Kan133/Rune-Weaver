## 二、核心困境：feature-assembler 是一座断桥

### Pattern 库存在，但 Assembler 不认识

| 功能 | Pattern 模板是否存在 | Assembler 是否支持 |
|------|---------------------|-------------------|
| `weighted_pool` (data_system) | ✅ `patterns/systems/data_system/weighted_pool/` | ❌ 无 `assembleDataSystem()` |
| `selection_modal` (ui) | ✅ `patterns/ui/selection_modal/` | ❌ 不识别 `type === 'selection_modal'` |
| `abilityEffect.custom` | ⚠️ 部分存在 | ❌ 总是 fallback 到 `dash+invul` |
| `ruleEngine` | ❌ 无模板 | ❌ 无运行时 |
| `stat_bonus` modifier | ❌ 无模板 | ❌ 无 assembler |

### 根本原因
`feature-assembler` 当前是**硬编码的 if-else 堆砌器**，而不是设计意图中的 **Blueprint → Pattern 动态路由器**。它只认识：
- `key_hint`
- `resource_bar`
- `custom_resource` (mana_like)
- `modifiers` (pattern-based)
- `abilityEffect`（但只会生成 dash+invul）

一旦用户需求稍微超出这个范围（如天赋抽取、卡池、装备三选一），Assembler 就会失效或生成错误的代码。

## 三、Gap 系统是用来做什么的？

### 设计意图
项目的原始架构是四阶段：
```
NL → LLM Blueprint (B1) → LLM Gap Fill (B2) → Assembler (C) → Wiring (D)
```

`llm-context` 包确实会扫描 `patterns/` 下的 `gap_regions.json`，生成 `gapInstructions`（即针对模板中特定区域的 LLM 填充提示）。例如：
- `[GAP:dash_params]`：让 LLM 填写冲刺运动的数学公式
- `[GAP:custom_effect]`：让 LLM 填写某个天赋的特殊效果逻辑

### 现实状态
**B2 步骤在 `pipeline.ts` 中从未被真正实现。**
- `gapInstructions` 被计算出来、被保存，但**从来没有被发回给 LLM**。
- `template-engine` 的 `renderTemplate()` 只是一个纯 Handlebars 字符串替换，**不识别 `[GAP:xxx]` 标记**，也不会等待 LLM 返回填充内容。
- 结果：模板里的 GAP 占位代码被原封不动地保留在生成文件中。

### 我们的决定
**今天先不恢复 Gap 系统。**
原因：
1. 工程量大，需要重构 `pipeline.ts` 的调用链、设计 GAP 返回内容的替换协议。
2. 当前天赋抽取需求的效果非常规则化（权重、颜色、属性加成），**完全可以由 Assembler 直接注入模板上下文**，不需要 LLM 创造性填充。
3. 对于未来的复杂天赋（如"闪避时反击"），我们可以在模板中预留 `// [EXTEND: custom_effect]` 注释，作为后续引入 Gap 系统的锚点。

## 四、重构方向（已确认）

### 1. feature-assembler 路由器重构
将 `assembleFeature()` 从硬编码 if-else 改为**按抽象能力动态路由**：

```
Blueprint Feature          → Assembler Function           → Pattern Template
─────────────────────────────────────────────────────────────────────────────
abilityEffect              → assembleAbilityEffect()      → 动态路由（dash/empty/custom）
modifiers[]                → assembleModifiers()          → metadata.json 指向的模板
customResource             → assembleCustomResource()     → mana_like / charge_like
customInput                → assembleCustomInput()        → 硬编码按键绑定（足够通用）
uiComponents.key_hint      → assembleKeyHint()            → 硬编码（足够通用）
uiComponents.selection_modal→ assembleSelectionModal()    → patterns/ui/selection_modal/
uiComponents.resource_bar  → 跳过（由 customResource 处理）
dataSystem.talent_pool     → assembleDataSystem()         → patterns/systems/data_system/weighted_pool/
dataSystem.card_pool       → assembleDataSystem()         → 同上（共用 weighted_pool）
ruleEngine                 → assembleRuleEngine()         → 轻量运行时模板（新）
```

### 2. Talent 代码组织模式
**不按具体天赋拆 16 个文件，而是采用注册表/工厂模式：**

```ts
// modules/talent_manager.ts
export interface ITalentEffect { (hero: CDOTA_BaseNPC): void; }
const TALENT_REGISTRY: Record<string, ITalentEffect> = {};
export function registerTalent(id: string, effect: ITalentEffect): void { ... }
export class TalentManager { applyTalent(playerId, talentId) { ... } }
```

对于当前 16 个简单占位符天赋，全部内联到一个文件：
```ts
// talents/talent_draw_pool.ts
registerTalent('r_001', hero => hero.ModifyStrength(10));
registerTalent('r_002', hero => hero.ModifyStrength(10));
// ... 共 16 条
```

**为什么这样做？**
- 单文件 switch 或 16 个独立文件对于"纯属性加成占位符"都过于冗余。
- 注册表模式是未来扩展的基础：当用户后续加入复杂天赋时，只需要继续 `registerTalent`。
- 500+ 天赋时，可以按稀有度拆成 `talents/rare.ts`、`talents/epic.ts` 等，避免文件过大也避免文件爆炸。
- Lua 运行时性能和 TS 编译性能都不会因为"一个文件里有很多注册表条目"而下降。

### 3. 新增内容清单
- `assembleSelectionModal()`：渲染 `selection_modal.tsx.template` + `.less.template`
- `assembleDataSystem()`：把 `talent_pool` 映射到 `weighted_pool`，注入 entries/weights
- `assembleRuleEngine()`：生成事件监听 → 抽取 → 发送弹窗 → 接收选择 → 应用属性的完整链路
- `abilityEffect` 智能路由：当 `effects.type === 'custom'` 且与 UI/数据系统配合时，不再无脑生成 dash+invul
- 可选：`patterns/modifiers/stat_bonus/server.ts.template`（如果 rule engine 直接调用 `ModifyStrength` 则可以暂不需要）

### 4. 不做的内容
- **不恢复 Gap 填充（B2）**：当前需求不需要 LLM 创造性填充，规则化数据足够。
- **不引入统一定位枚举系统**：之前讨论过，认为会限制用户"往左挪 20px"的微调需求。继续使用具体像素值，未来通过自然语言 refine 工具来微调。
