# Dota2 Assembler 第一阶段范围定义

## 1. 文档目的

本文档明确 Dota2 Assembler **第一阶段**的真实落地范围。

它不是 assembler 的完整功能规格，而是回答：

- 第一阶段到底负责什么
- 第一阶段**不负责**什么
- 与 `AssemblyPlan` 如何对齐

## 2. 第一阶段负责范围

### 2.1 生成骨架文件

Assembler 第一阶段只负责生成三类骨架文件：

| 类型 | 输出路径 | 内容 |
|------|----------|------|
| Server 骨架 | `game/scripts/src/rune_weaver/generated/server/{featureId}.ts` | Pattern 组合后的服务端逻辑骨架 |
| Shared 骨架 | `game/scripts/src/rune_weaver/generated/shared/{featureId}.ts` | 类型定义和常量 |
| UI 骨架 | `content/panorama/src/rune_weaver/generated/ui/{featureId}.tsx` | UI 组件骨架 |

> Phase 1 明确不支持 `update`。当前只定义生成骨架内容与后续交给 Write Executor 的写入边界。
> 如文档中出现 `update` 一词，应理解为未来阶段语义，而不是当前可执行动作。

### 2.2 骨架定义

"骨架"意味着：

- 包含完整的 import/export 语句
- 包含函数/组件定义签名
- 包含必要的类型注解
- 包含 TODO 标记的待实现逻辑

"骨架"**不意味着**：

- 完整的业务逻辑实现
- 可直接运行的最终代码
- 复杂的代码生成逻辑

### 2.3 示例骨架输出

**Server 骨架示例** (`rw_dash_q.ts`):

```typescript
// Generated by Rune Weaver
// featureId: rw_dash_q
// blueprintId: dash_ability_xxx

import { registerAbility, AbilityKind } from "../../../rune_weaver/runtime";

/**
 * Dash Q 技能实现
 * Pattern: effect.dash + input.key_binding
 */
export function registerRwDashQ(): void {
  // TODO: 实现冲刺技能注册
  registerAbility("rw_dash_q", {
    kind: AbilityKind.DASH,
    keyBinding: "Q",
    // 参数来自 Blueprint
  });
}
```

**UI 骨架示例** (`rw_dash_q.tsx`):

```tsx
// Generated by Rune Weaver
// featureId: rw_dash_q
// blueprintId: dash_ability_xxx

import React from "react";

/**
 * Dash Q 按键提示组件
 * Pattern: ui.key_hint
 */
export function RwDashQHint(): React.ReactElement {
  // TODO: 实现按键提示 UI
  return (
    <Panel className="rw-key-hint">
      <Label text="Q - 冲刺" />
    </Panel>
  );
}
```

## 3. 第一阶段不负责范围

### 3.1 不负责任意旧文件 merge

Assembler **不**负责：

- 读取宿主已有文件内容
- 与现有代码进行智能合并
- 处理用户修改过的生成文件

### 3.2 不负责自由代码生成

Assembler **不**负责：

- 基于 LLM 的自由代码生成
- 复杂的业务逻辑推导
- 跨文件的复杂代码编排

### 3.3 不负责任意宿主文件改写

Assembler **只**写入 RW 命名空间目录：

```
✅ 允许: game/scripts/src/rune_weaver/generated/**
✅ 允许: content/panorama/src/rune_weaver/generated/**

❌ 不允许: game/scripts/src/examples/**
❌ 不允许: game/scripts/src/modules/ (除 index.ts inject_once)
❌ 不允许: content/panorama/src/hud/ (除 script.tsx inject_once)
```

## 4. 与 AssemblyPlan 对齐

### 4.1 输入：AssemblyPlan

```typescript
interface AssemblyPlan {
  blueprintId: string;
  selectedPatterns: SelectedPattern[];
  writeTargets: WriteTarget[];
  bridgeUpdates: BridgeUpdate[];
}
```

### 4.2 输出：GeneratedArtifact[]

```typescript
interface GeneratedArtifact {
  /** 生成的文件路径 (相对宿主根) */
  filePath: string;
  /** 文件内容 */
  content: string;
  /** 所属 feature */
  featureId: string;
  /** 来源 pattern */
  patternIds: string[];
  /** 内容类型 */
  contentType: "skeleton" | "full";
}
```

### 4.3 处理流程

```
AssemblyPlan
  ↓
[按 target 分组]
  ↓
[为每个 target 选择模板]
  ↓
[填充骨架参数]
  ↓
[输出 GeneratedArtifact[]]
```

## 5. 模板选择策略

### 5.1 当前支持的 Pattern 组合

| Pattern 组合 | 输出骨架 | 说明 |
|--------------|----------|------|
| `input.key_binding` + `effect.dash` | server + ui | 按键冲刺技能 |
| `rule.selection_flow` + `ui.selection_modal` | server + shared + ui | 选择系统 |
| `resource.basic_pool` + `ui.resource_bar` | server + shared + ui | 资源系统 |

### 5.2 模板选择规则

```typescript
function selectTemplate(patterns: SelectedPattern[]): TemplateKind {
  const ids = patterns.map(p => p.patternId);
  
  if (ids.includes("effect.dash")) return "dash-ability";
  if (ids.includes("rule.selection_flow")) return "selection-system";
  if (ids.includes("resource.basic_pool")) return "resource-system";
  
  return "generic";
}
```

## 6. 接口定义

### 6.1 Assembler 接口

```typescript
// adapters/dota2/assembler/index.ts

export interface AssemblerOptions {
  hostRoot: string;
  featureId: string;
  blueprintId: string;
}

export interface AssembleResult {
  success: boolean;
  artifacts: GeneratedArtifact[];
  errors: string[];
}

/**
 * 执行 AssemblyPlan 生成骨架文件
 */
export function assemble(
  plan: AssemblyPlan,
  options: AssemblerOptions
): AssembleResult;
```

### 6.2 使用示例

```typescript
const result = assemble(assemblyPlan, {
  hostRoot: "D:\\test1",
  featureId: "rw_dash_q",
  blueprintId: "dash_ability_xxx",
});

if (result.success) {
  for (const artifact of result.artifacts) {
    console.log(`Generated: ${artifact.filePath}`);
  }
}
```

## 7. 与 Write Executor 的关系

Assembler 只负责**生成内容**，不负责**写入文件**。

写入操作由 Write Executor 负责：

```
Assembler → GeneratedArtifact[] → Write Executor → 宿主文件系统
```

## 8. 验收标准

- ✅ 能根据 AssemblyPlan 生成三类骨架文件
- ✅ 生成的骨架包含完整的类型签名和 TODO 标记
- ✅ 只写入 RW 命名空间目录
- ✅ 不执行任意旧文件 merge
- ✅ 与 Write Executor 有清晰边界

## 9. 当前限制

| 限制 | 说明 |
|------|------|
| 只生成骨架 | 不包含完整业务逻辑实现 |
| 模板数量有限 | 只支持预定义的 Pattern 组合 |
| 无智能合并 | 不处理与现有代码的冲突 |
| 单宿主 | 只支持 dota2-x-template |

## 10. 后续扩展方向

Phase 2 可能扩展：

- 更完整的业务逻辑生成
- Pattern 参数验证和转换
- 代码片段复用机制

但 Phase 1 必须守住：**骨架生成，受控目录，清晰边界**。
